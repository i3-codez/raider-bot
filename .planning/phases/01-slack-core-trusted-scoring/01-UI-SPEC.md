## Modal Contract

Use a Slack modal for `/raid` with these exact inputs and labels:

| Order | Field | Element | Rules | Source |
|------|-------|---------|-------|--------|
| 1 | `Post URL` | `url_text_input` | Required. `https` only. Accept X/Twitter URLs only in Phase 1. `focus_on_load: true`. | `RAID-01`, Slack `url_text_input` docs, X-first scope from `.planning/PROJECT.md` |
| 2 | `Client name` | `plain_text_input` | Required. Single line. Trim whitespace. 2-60 visible characters. | `RAID-01`, default validation |
| 3 | `Platform` | `static_select` | Required. One enabled option: `X`. Do not show unsupported future platforms in Phase 1 UI. | `RAID-01`, X-first scope from `.planning/PROJECT.md` |
| 4 | `Published at (optional)` | `datetimepicker` | Optional. Leaving it blank is valid and must not block submission. | D-01, Slack `datetimepicker` docs |

Modal copy rules:

- Title: `Create raid`
- Submit: `Create raid`
- Close: `Close without posting`
- Intro helper copy: `Post a raid target for the team. Published time is optional.`
- Published-time hint: `Leave blank if unknown. Raider Bot will use Slack post time and mark the raid as approximate.`
- Validation copy for future timestamp: `Published time can't be in the future.`
- Validation copy for unsupported URL: `Use a valid X post URL.`

## Raid Message Contract

Build every raid message with this block order. Keep the live message to 8 blocks or fewer.

1. Top-level `text` fallback for accessibility that summarizes client, platform, post URL, timing confidence, reaction legend, and timing-window emphasis.
2. `header` block: `Raid live: {client_name} on X`
3. Primary `section` block with the post URL as a standalone first line and the platform/client metadata beneath it.
4. Timing status row:
   - High confidence: `Published: {timestamp} ET`
   - Low confidence: `:warning: Published time not provided. Using Slack post time, so timing-based scores are approximate.`
5. `divider`
6. Reaction legend section with exactly four items:
   - `:heart: Like`
   - `:speech_balloon: Comment`
   - `:repeat: Repost`
   - `:memo: Quote post`
7. Timing rules section listing the five windows in descending order: `0-10m`, `10-20m`, `20-30m`, `30-60m`, `60m+`
8. Footer/context note: `One score per person per action type. Remove your reaction to undo that action.`

Message composition rules:

- Primary focal point: the standalone post URL row. The header is secondary orientation only.
- The post URL must appear above the legend and above the timing rules.
- Timing confidence gets its own row. Do not bury approximate timing in footer copy.
- Reaction legend and timing rules must both be visible in the initial message body. Staff should not need `/raiderhelp` to participate correctly in Phase 1.
- Do not add interactive buttons to the raid message.
- Do not use more than one emoji in the header line.
- Use plain language. Never abbreviate timing confidence to `low conf` or similar shorthand.

Sources: `RAID-05`, `RAID-06`, `ENG-01` through `ENG-05`, D-02, D-03, `.planning/research/PITFALLS.md`.

## Scoring Display Contract

The UI must display the scoring model from one canonical source, not from duplicated strings spread across handlers.

- Reaction labels come from the same centralized emoji map used by the scoring engine.
- Timing windows come from the same centralized timing config used for point calculation.
- If exact point values are shown, they must be rendered from canonical scoring config at message-build time.
- If point values are not shown in the raid message, the message must still state that earlier windows score more and list all five windows explicitly.
- Any future correction to `published_at` must update the dedicated timing row without changing the legend layout.

Source: Phase discretion in `.planning/phases/01-slack-core-trusted-scoring/01-CONTEXT.md`, `ENG-04`, `.planning/research/PITFALLS.md`.

## Interaction Rules

- `/raid` opens the modal immediately after the slash command. Do not require operators to memorize argument syntax in Phase 1.
- Successful submission posts the raid in the configured channel and returns an ephemeral confirmation with the channel name and timing confidence.
- Unsupported emoji reactions are ignored silently. Do not post channel noise for non-scoring reactions.
- Re-adding the same supported action after it is already active must not create a second visible claim or a second score.
- Removing a supported reaction reverses that action quietly. Do not announce score reversals publicly in the channel.
- Reactions on non-raid messages are ignored silently.
- If the raid was created without `published_at`, every staff-facing surface for that raid must use the word `approximate`.

Source: `RAID-01`, `RAID-06`, `ENG-02` through `ENG-05`, D-01 through D-04.

---

## Accessibility and Content Rules

- Include a complete top-level message `text` fallback for screen-reader users instead of relying on Slack to synthesize content from blocks. Source: Slack Block Kit accessibility guidance.
- Use explicit timezone labels. Show `ET` in every human-readable timestamp shown to staff.
- Keep each legend label to one noun phrase only: `Like`, `Comment`, `Repost`, `Quote post`.
- Keep the most important action path tappable with one link: the raid target URL.
- Do not depend on color alone for confidence or error states. Use status words plus emoji.

Sources: Slack Block Kit docs, `.planning/PROJECT.md`, `.planning/STATE.md`.
